Adapting: 从monolithic到microservice升级

技术一旦出现将不再回头。新技术，新需求永远都在变化，adapting是不变的关键词。

计划

对于大型企业的诸多项目啦来说，从一体式结构拆成微服务似乎变成了一项必备项。如何稳定平滑的做技术升级，处理好新技术升级带来的复杂度挑战，并最终让各级用户感知到升级带来的畅快感。好比我们从木板床变成席梦思，用户更舒适了。当然，升级也有很多个维度，从总体的全公司技术方向，部门的基础平台，调度，管理在到团队的具体项目升级。 这里只有一些做升级的过程中对于一个具体的小项目纬度的拆分的记录。作为储备。

首先，legacy monolithic application是很不好拆分的。从分层的概念上来说，application layer code， domain logic， data layer 全都是在一起的，其中商业逻辑，最重的部分，有着很多千丝万缕的依赖。看上去无法拆分。可是俗话说实践是检验真理的唯一标准。老项目是实实在在的在线上工作的，经过了在线检验。想要把他完全重写成现代化规范，步子太大，风险太高，成本太多。

从最本能的生活想法出发：
就好像是好拆汽车，当看见乱七八糟的复杂零件的时候，心里自然而然会有一个规划：

  1. 有大概哪几个大块。因为汽车零件我们可以肉眼看到，大脑自动就处理了。

  2. 先从最简单的，小的，可以动手的地方开始拆。把小部件拆出来以后给大部件留出挪移的空间。

  3. 拆除来的部件还的是能单独运行的，不能拆坏了。

  4. 各个部件之间以后就靠电线来合作交流。

  5. 有些是在一体化的零件拆不了的，只能用他们，可是螺丝位置又不完全匹配，只能是用一些桥接手段来放置。

  6. 想着点以后的新车模，这些引擎，零部件还得往那边挪。

照着这个模版。在聊聊项目的问题：

1.老项目需要做ui升级。老ui也要接着用。

2.ui升级一定意味着数据展示会有变化，用户操作会有变化。

3.而且不仅仅是ui升级，还需要集成新的第三方系统，并在ui上合理展示。

步骤

首先做了两个简单的处理步骤：

前端在做升级的过程中，发现老的api极度不友好，不规范 -> 接口全部重新标准化定义，做api版本隔离v1，v2。老的接口老版本继续用，新的接口收到请求后转发到老接口。最小化后端改动。不影响老项目。 -> proxy layer

前台要求简化，增加，全新，数据。 -> 新接口里接受到的数据预处理，得到老数据后的post处理 -> anti-corruption action in proxy layer

有了这两步，至少前端可以联调开发，不会block前端的进度。

可是这个proxy是没有任何商业逻辑的。proxy只能止渴。所以老项目的分拆还是避免不了，因为还要集成第三方系统。想象一下如果不分拆：新的逻辑无处落脚。写到一团乱麻的老项目里只会增加未来分拆的难度。安全落地新的集成也需要同样的复杂依赖分析。

看了些网上的大致经验，也正好匹配前面拆汽车零部件的步骤：

  1. 先refactor，划分出清楚的商业逻辑关系。
  2. 先拆出一个明显的简单的部分。比如用户，权限，monitor等
  3. 老代码能不动就不要动，但是可以新增一些胶水代码在旧/新的服务上。做anti-corruption layer。
  4. 数据库千万不要去做规范化整理。代价太大。
  5. 不宜拆的模块划分，并做好优先级排序。
  6. 新的集成一定要去新的service里面。不要放在老项目里了。因为以后要转到microservice。
  7. orm要注意，看看id生成会不会有冲突在云环境里。
  8. 最好是每个service有自己的存储。

开始分析项目：

  • 各个layer之间有清晰的职责划分。application层只做与client的交互问题，domain层做商业逻辑，data layer只做数据。
  • 可以从application/db table 双向出发，找到简单清晰的关系清楚的。因为商业逻辑不管多复杂，最初都始于易用的ui，终于都落表。
  • 通过static code analysis工具也可以发现边界清楚的模块。
  • 画出aggregates/entity/value-object 树组成。root必须是一个entity。在包含多个value-objects或多个aggregates。保持外界只有root的reference。root与内部对话。value copy to outside instead of pass inner object reference to outside.
  • 通过factory 新建arrgregates/entity, 通过repo获取/存储已经存在的（db）
  • 制定出event/event handler
  • 分析所属的specification/strategy
