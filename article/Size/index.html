怎么确定一个pool的大小上限？

pool的大小上限很重要，因为他可以让你尽可能的提高系统利用率。在了解你的系统的能力，应对系统峰值或者系统自身的瓶颈有一些线索。说实话，在没有任何测量与测试的情况下，怎么找到一个池的大小是有点盲人摸象，也不会找到系统的最优点。但是却有一套比较实用的理论去帮助我们找到一个起始点。

在java里面有一些封装好的thread pool。 看一下代码：

注意到SynchronousQueue，每一个thread都会在这里面排队。当工人不够时候，会增加一个新的工人到这里面排队。这里就会出现两种情况 ->
1. 活太多，一直在增加工人，厂子爆了。out of memeroy。
2. 活太少， 工人没少，全部的人都在等活， thread starvation。
为避免这种情况的出现，我们需要找到一个大小上限。就算没有活/活太多，也不会让机器崩溃。

起始点

首先不要忘记审视自己的系统和硬件。 比如：
数据库连接，你的池子里有一万个工人，但是你连接的数据库能不能扛住一万人同时找它呢。
一些第三方系统api根本支持不了或者有限制，这时候工人在多没用。

最基本的，我们需要知道我们有多少个cpu。

一般环境下是这样的，但是对于容器来说，java8 不能准确的获取容器能使用的cpu。Java10+ 有特定的修复。reference: java in container

在知道cpu核数的情况下，一般的公式如下：

数量 = 核数 （1+ io时间/计算时间）*

举个例子， 2cpu io时间400ms 计算时间50ms -> 16 in thread pool 公式里的 （io时间/计算时间）简称为阻塞时间。系数接近0 的时候，系统每个cpu都用满了，这就是最大的thread 数量。如果系数越大 说明系统越多的是在做io，线程数量则可以更多。

但是应用当中有可能有多个线程池。每个线程池对应的io/cpu time是不同的。所以公式可以更新为：

数量 = 核数 cpu使用率（0-1) (1+ io时间/计算时间)**

另外一个概念： little’s law 可以帮助我们知道rps数据。
l = r*w
l: 同时处理的请求量
r: rate per second
w: 处理时间
-> 16/0.45 = 35 request per second

总结

这些公式是一个好的起始点，在通过审视子系统的连接处理能力，三方处理能力 和一些测试，才能发现真正的最优解。
