Anti Pattern At Physcial Level

Numerical Precision

Key Value
Target save high precision numbers
Anti-Pattern use “Float”
Normal-Pattern use ‘Numeric’/ ‘Decimal’

ENUM

Key Value
Target Save a ENUM type
Anti-Pattern Define a Enum type in MYSQL or set condition like column in (‘a’,‘b’,‘c’)
Normal-Pattern use extra static table to get extensibility.

Example:
define status in (‘new’,‘in-progress’,‘finished’), however in new requirement comes and we have to alter table to add a new type like 'duplicated 'in this status column.

id status
1 new
2 in-progress
3 finished

->

id status_id
1 1
2 2
3 3
4 4
status_id status_name
1 new
2 in-progress
3 finished
4 duplicated

Phantom Files

Key Value
Target integrate file system to store files and store relationships within DB.
Anti-Pattern Bad disaster recovery, unforced transaction, unforced backup.
Notice when disaster happens, cant reply on DB backup only, cause the file is seperately stored in file system.
Normal-Pattern try to use ‘BLOB’ in extra table.

Too Many Index

Key Value
Target To optimize the performance, Build Index without reasoning and Analysis, Planning as well as maintainess.
Anti-Pattern build too many indexes
Notice check the selectivity. COUNT(DISTINCT status) /COUNT(status). if the rate is high, then the index effectiveness is low. it’s like tons of duplicates in dictionary, we have to check these duplicates one by one, then the index’s function is lost.
Normal-Pattern tools to MENTOR(Mesure,Explain,Nominate,Test,Optimize,Rebuild)
  • Mesure -> Slow query log.
  • Explain -> Why it’s slow and where did time spend.
  • Nominate -> find out queries without index, find out coverage rate of queries with index
  • Test -> After build index, test the performance increase.
  • Optimize -> Cache.
  • Rebuild -> Balance the index tree.

Anti Pattern On Query

NULL Value

NULL can not be treated as normal value.

NULL=NULL unknown
NULL+10 unknow
Column_name = null unkown.

Try to use is NULL. or forbiden NULL at column condition.

Groups

Group by can only be used on single value rule.

Query:
select product_id, max(date_reported) from product group by product_id; -> valid
select product_id, max(date_reported), bug_id from product join bugproducts using bug_id group by product_id; -> invalid
break the single value rule, cause DB cant ensure the bug_id is single value.

Solution:

  1. get rid of uncertain column
  2. use sub select
  3. use temp table
  4. use join

Random

Order by random and select first column. -> easy to understand but it’s slow, and waster of work.

Solution:

  1. direct find a random key from begin to end if key is continuous.
  2. find next max value when key is not continous used.
  3. read all keys into program and random select one.
  4. use curse to random jump to one row.
  5. use ‘sample’ clause.

Full Text Search

Use Lucene, even though some DB support some kind of similar search

Select Chain

Sometimes, Developer always try to solve the requirement in one query, like:
“SELECT COUNT(bp.product_id) AS how_many_products,
COUNT(dev.account_id) AS how_many_developers,
COUNT(b.bug_id)/COUNT(dev.account_id) AS avg_bugs_per_developer,
COUNT(cust.account_id) AS how_many_customers
FROM Bugs b JOIN BugsProducts bp ON (b.bug_id = bp.bug_id)
JOIN Accounts dev ON (b.assigned_to = dev.account_id)
JOIN Accounts cust ON (b.reported_by = cust.account_id)
WHERE cust.email NOT LIKE '%@example.com’
GROUP BY bp.product_id;”

But, it’s hard to understand and maintain. try to seperate and go step by step will be better.
