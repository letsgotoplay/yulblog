从平房开始看 - Spring boot admin(一)

Spring boot admin（sba），从简写的名字就可以看出这是一个傻瓜，结构简明的好平房。 所以就从这里开始看吧。
spring boot admin，有什么功能呢？
一个中心化的展示器server。会通过client项目里面的Spring actuator的endpoint获取client的信息并可以发出一些Spring actuator允许的一些指令，远程ui界面操控client项目。 所以功能流程图应该是：

clientA – register – > {serverA}
clientB – register – > {serverA}

ServerA – request all clients endpoints --> UI display

UI control – request controll --> server --> client changes

功能简单。就⬆️三步。

想清楚了功能流程图， 接下来就是拆解关键字，划分出功能领域模型：

client

client是由spring-boot-admin-client组成。功能领域只有一个： registration。这个lib主要提供registration功能给client项目。所以自然会有一个功能开关，其实也就是常见的spring conditional configuration。
所以package也简单名了：

com…config包 //实现项目的自动化/可定义条件的配置
com…registration包 //实现项目的注册功能

由于是基于Spring boot的lib包。所有的object都是用@bean来实现实例化（@bean/@component 的区别），constructor构建。如果属性功能开关开启，auto-config类基于条件被Spring 管理并自动注册全部的功能相关的bean.

如果是reactive 环境用web client 发送请求到server注册。

server

然鹅， 我们知道如果服务有服务发现中心的，我们是可以问服务发现中心要到所有的服务地址的。比如eureka。所以client的lib的注册机制其实server自己去问eureka也能拿到。

在来看看server吧：

  1. web layer -> header provider + controller + proxy -> API
  2. shared kernel -> util + serializer
  3. domain -> aggregate with entity root + value object + events object
  4. conditional config -> feature switch + bean init
  5. service -> info/status update + instance registry + event_listener
  6. eventstore -> event storeage as repo
  7. notifier+alert feature

所以重点是service 里面的update 以及event listener里面的功能实现。这个稍后再详细说明。
首先admin正常启动，需要@enableadminserver注解。这个注解和@Springboot注解的功能是一样的。不过有些许不同的是，@admin会import markerconfig 并实例化一个marker object。 所有的其他的condition config都会基于这个marker object是否存在。相当于一个总开关。下面可能会有一些可配置的子开关在property file里设置。
