接着来看看spring admin server。在这之前先了解一些基本的概念和应用技术。

event store pattern/event source pattern

传统的pattern是什么呢？ crud数据处理。多数基于transaction。但是传统的模式也有他的限制。

  1. 颗粒度大。 正如锁机制一样，为大颗粒度操作而生。
  2. crud直接更新数据存储，操作数量级有限。responsiveness不够高。所以对于大量并行操作并不十分友好
  3. 所有操作历史丢失。没法追溯历史操作，在不额外做audit的情况下。

而event store pattern：

数据处理操作全部转化成event并且append only到event store。 event store负责存储持久这些event，并且可以发送这些event到下游监听者。解耦很好，上游只需发送event，下游只管处理event携带的操作。可以有追踪历史。可以增加资源使用度，因为下游可以串行接受消息，中心思想和reactor有点相似。

reactive programming

有个很好的例子：买咖啡。
阻塞sync： 传统排队买咖啡，没拿到咖啡就一直等着。
非阻塞async：下单买咖啡，继续干正事，收到消息咖啡送到了，取咖啡。

所以非阻塞async是不会等待的，人（thread）可以干别的事情。当咖啡好的时候会有一个通知机制。所以是有时间观念的。咖啡送达的快慢，没等到就退单，阻塞sync没有时间观念，死等，一个人只能干一件事儿。

三个基本概念： publisher, subscriber, subscribe()

flux,mono -> publisher。
从分类上来说有 静态工厂方法生产publisher实例，operator方法如何处理数据，异常处理方法如何容错，线程方法如何分配工作，subscibe方法下游反馈。 publisher在没有subscribe之前不会发送数据。相当于工厂流水线，设备流程都准备好了，就等下游的subscriber一声令下，消息经过流水线送到下游手中。

server流程

前期分析了一些基本domain object，一般的会有domain object a + builder for a + factory to use builder to create domain a.
然后在admin server里面 instance是一个巨大的aggregate.里面属性众多。包装了许多子object。在这里用到了什么方法呢？ 小的子object依然使用前面的套路。大型aggregate使用的是constructor+ 自身的apply（event）(return new instance(…))。

在展开admin server的流程图：

entity: instance repository -> get event store -> get event -> instance apply event -> get instance publisher.//数据存储
service: endpoints trigger -> info updater -> event -> event store -> repository //数据来源
controller: request filters -> reactive controller -> registry -> repository -> reactive streams //数据展示
